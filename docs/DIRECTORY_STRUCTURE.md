# Directory Structure

## Overview
The extension organizes build outputs in a clean, hierarchical structure under `.clash/` in your workspace root.

## Structure

```
.clash/
└── {ModuleName}.{FunctionName}/          # Fully qualified function name
    ├── 01-haskell/                       # Stage 1: Clash wrapper generation
    │   └── ClashSynth_FunctionName.hs    # Generated wrapper module
    │
    ├── 02-verilog/                       # Stage 2: Clash compilation
    │   └── ModuleName.topEntity/
    │       ├── function_name.v           # Main Verilog file
    │       ├── function_name_types.v     # Type definitions (if generated)
    │       ├── helper_module.v           # Helper modules (if any)
    │       ├── clash-manifest.json       # Clash metadata
    │       └── ...                       # Other Clash outputs
    │
    ├── 03-yosys/                         # Stage 3: Yosys synthesis
    │   ├── function_name_synth.v         # Synthesized Verilog netlist
    │   ├── function_name.json            # JSON netlist (for nextpnr)
    │   ├── synth.ys                      # Yosys synthesis script
    │   ├── yosys.log                     # Complete Yosys output
    │   ├── statistics.txt                # Formatted statistics report
    │   └── synthesis_stats.txt           # Raw synthesis statistics
    │
    └── 04-nextpnr/                       # Stage 4: Place & route + bitstream
        ├── function_name.config          # Textual configuration
        ├── function_name.bit             # FPGA bitstream (ready to program!)
        ├── nextpnr.log                   # Complete nextpnr output
        ├── ecppack.log                   # Bitstream generation log
        ├── timing.txt                    # Formatted timing analysis
        └── utilization.txt               # Formatted resource utilization
```

## Example: plusSigned Function

For a function `plusSigned` in module `Example.Project`:

```
.clash/
└── Example.Project.plusSigned/
    ├── 01-haskell/
    │   └── ClashSynth_PlusSigned.hs
    ├── 02-verilog/
    │   └── ClashSynth_PlusSigned.topEntity/
    │       ├── plus_signed.v
    │       └── clash-manifest.json
    ├── 03-yosys/
    │   ├── plus_signed_synth.v
    │   ├── plus_signed.json
    │   ├── synth.ys
    │   ├── yosys.log
    │   └── statistics.txt
    └── 04-nextpnr/
        ├── plus_signed.config
        ├── plus_signed.bit              ← Ready-to-program bitstream!
        ├── nextpnr.log
        ├── ecppack.log
        ├── timing.txt
        └── utilization.txt
```

## Stage-by-Stage Explanation

### 01-haskell: Wrapper Generation
Contains the generated Clash wrapper module with `Synthesize` annotations. This is auto-generated from your original Haskell function and includes:
- Import of original module
- Type-safe wrapper definition
- `Synthesize` pragma with proper naming
- Port name generation

### 02-verilog: Clash Compilation
Contains Verilog generated by the Clash compiler. May include:
- **Main module** (function_name.v): The primary Verilog implementation
- **Type definitions** (function_name_types.v): Custom type implementations
- **Helper modules**: Additional Verilog modules for complex designs
- **Manifest**: JSON metadata about the compilation

### 03-yosys: Synthesis
Contains results from Yosys synthesis:
- **Synthesized netlist** (_synth.v): Optimized Verilog for target FPGA
- **JSON netlist** (.json): Machine-readable format for nextpnr
- **Synthesis script** (synth.ys): The Yosys script that was executed
- **Logs**: Complete tool output for debugging
- **Statistics**: Cell counts, wire counts, resource usage

Key reports:
- `statistics.txt`: Formatted report with cell type breakdown
- Contains: cell count, wire count, chip area, cell types sorted by usage

### 04-nextpnr: Place & Route + Bitstream
Contains final FPGA implementation results:
- **Config file** (.config): Textual FPGA configuration
- **Bitstream** (.bit): Ready-to-program binary (upload this to FPGA!)
- **Logs**: Complete output from nextpnr and ecppack
- **Reports**: Formatted timing and utilization analysis

Key reports:
- `timing.txt`: Maximum frequency, critical path, slack (PASS/FAIL)
- `utilization.txt`: LUT/Register/BRAM/DSP/IO usage with visual bars

Example timing.txt:
```
Timing Analysis Report
============================================================
Maximum Frequency:     125.30 MHz
Critical Path Delay:   7.981 ns
Setup Slack:           0.234 ns [PASS]
Hold Slack:            0.145 ns [PASS]

Overall Status:        ✓ CONSTRAINTS MET
```

Example utilization.txt:
```
Resource Utilization Report
============================================================
LUTs              245 / 24000  (  1.02%)  [█░░░░░░░░░░░...]
Registers         178 / 24000  (  0.74%)  [░░░░░░░░░░░░...]
BRAM/EBR            0 / 56     (  0.00%)  [░░░░░░░░░░░░...]
IO                 18 / 197    (  9.14%)  [███░░░░░░░░░...]
```

## Benefits of This Structure

### Isolation
Each function synthesis is completely isolated in its own directory. You can:
- Synthesize multiple functions without conflicts
- Compare implementations of different functions
- Archive/share specific function implementations
- Clean up individual functions without affecting others

### Organization
Numbered stages (01-04) provide:
- Clear progression through the build pipeline
- Easy navigation (stages always in same order)
- Immediate understanding of what each directory contains
- Alphabetical sorting matches build order

### Traceability
Complete build history for each function:
- All intermediate files preserved
- Full logs available for debugging
- Reports available for documentation
- Easy to reproduce exact build

### Clean Workspace
- Single `.clash/` directory to ignore in git
- No scattered build artifacts
- Easy cleanup: `rm -rf .clash/`
- Optional per-function cleanup: `rm -rf .clash/Module.Function/`

## Git Integration

Add to your `.gitignore`:
```gitignore
.clash/
```

This ignores all build outputs while keeping your source code clean.

## Navigating Build Outputs

### Find Your Bitstream
```bash
# For function 'plusSigned' in module 'Example.Project':
ls .clash/Example.Project.plusSigned/04-nextpnr/*.bit
```

### Check Timing
```bash
cat .clash/Example.Project.plusSigned/04-nextpnr/timing.txt
```

### Review Synthesis Statistics
```bash
cat .clash/Example.Project.plusSigned/03-yosys/statistics.txt
```

### Debug Issues
```bash
# Check Yosys output
less .clash/Example.Project.plusSigned/03-yosys/yosys.log

# Check nextpnr output
less .clash/Example.Project.plusSigned/04-nextpnr/nextpnr.log
```

### Compare Implementations
```bash
# Compare resource usage of two implementations
diff \
  .clash/Example.Project.plusSigned/04-nextpnr/utilization.txt \
  .clash/Example.Project.plusOptimized/04-nextpnr/utilization.txt
```

## Migration from Old Structure

If you have builds from the old `.clash-synth/` structure, you can safely delete it:

```bash
rm -rf .clash-synth/
```

The new structure is:
- More organized (numbered stages)
- More isolated (per-function directories)
- More complete (all logs and reports saved)

## Advanced Usage

### Archiving Successful Builds
```bash
# Archive a successful implementation
tar czf archives/Example.Project.plusSigned-v1.tar.gz \
  .clash/Example.Project.plusSigned/
```

### Cleaning Failed Builds
```bash
# Remove only failed builds (no bitstream)
for dir in .clash/*/; do
  if [ ! -f "$dir/04-nextpnr/"*.bit ]; then
    echo "Removing failed build: $dir"
    rm -rf "$dir"
  fi
done
```

### Batch Synthesis Reports
```bash
# Generate summary of all implementations
echo "Function,LUTs,Frequency"
for dir in .clash/*/; do
  name=$(basename "$dir")
  utilization=$(grep "^LUTs" "$dir/04-nextpnr/utilization.txt" | awk '{print $2}')
  timing=$(grep "^Maximum Frequency" "$dir/04-nextpnr/timing.txt" | awk '{print $3, $4}')
  echo "$name,$utilization,$timing"
done
```
